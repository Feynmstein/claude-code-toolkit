# Claude Code 核心约定速查

> 这不是给 Claude 读的——这是给你读的。每次接手新项目前花 2 分钟扫一遍。

---

## 一、项目组织

### 一个 project = 一个目的 = 一份 CLAUDE.md

- 工程项目和元认知项目必须分离
- CLAUDE.md 只服务于当前项目的单一目标
- 学习产出回流到 dedicated 的学习 project，不混在工程代码里

### 接手代码时，保留原版不动

- 重构在新目录中进行，原目录作为只读参考
- CLAUDE.md 中标注原代码路径和"只读"约定

---

## 二、CLAUDE.md 编写（四条铁律）

1. **写约束，不写教程**——Claude 不需要通用知识，需要这个项目的特殊规则
2. **写反模式，不写最佳实践**——"不要做什么"比"应该做什么"有效 10 倍
3. **短小精干**——控制在 80 行以内，每次会话都注入，越长越贵
4. **保持更新**——阶段变了立刻改，过时的 CLAUDE.md 比没有更危险

### CLAUDE.md 最小有效结构

```
项目目的 → 技术约束 → 架构约定 → 使用约定 → 当前阶段
```

### playout 实战的证据

playout CLAUDE.md 中 7 条核心约定，在最终代码中 100% 得到执行。

---

## 三、Plan Mode 决策

### 四条进入标准（全部命中才用）

1. 涉及多个文件的新功能或架构
2. 存在多种合理的技术方案
3. 需要用户在写代码前审批方向
4. 用户偏好会影响实现方向

### 不该用 Plan Mode

- 修 typo、加单函数测试、改一行代码
- 需求已经明确、无歧义的任务
- 纯探索性调研（先用 Explore agent 扫）

### Plan Mode 的局限性

- 无法覆盖实现细节
- 测试精力可能分配失衡（易测的模块测试多，核心逻辑测试少）
- 计划是静态的，中途偏离需退出重规划

---

## 四、测试信任链（三层防线）

### 三层防线模型

单元测试告诉你能通过 → 集成测试告诉你管线没断 → 可执行文档告诉你用户能跑通

```
第一层：单元测试 — 函数/方法逻辑正确性
第二层：集成测试 — 端到端管线：构造→导出→读回验证 ← 最容易被忽略
第三层：可执行文档 — 用户指南中的每个代码块都能复制即跑
```

### playout 的证据

- 273 个单元测试全部通过，但 Logo 加载一行就炸了（`parents[N]` 路径错误）
- GDS 往返发现 199 顶点限制导致多边形自动拆分——单元测试永远发现不了
- 写文档时跑真实代码发现了 2 个 Bug + 1 个 DXF 兼容性问题

### 落地要求

- **每个 IO 路径至少一个往返测试**（导出→读回，验证一致性）
- **文档中的示例代码必须可自动验证**（文档 = 被动集成测试）
- 新模块写完先写一个集成测试确立基线，再补单元测试

---

## 五、执行节奏

### 分阶段执行，不要一口吃成胖子

- 互相依赖的模块必须按阶段顺序推进（如 Phase 0→1→2→3）
- 每完成一个阶段跑一次测试
- **高风险改动必须独立迭代**，不要和其他修改混在一起

### 测试优先覆盖高风险区域

- 别被"每模块都要有测试"迷惑
- 按**风险**分配测试精力——领域逻辑 > IO 管线 > 胶水代码 > 简单函数

## 六、Session 管理三大纪律

### 1. 50% 上下文红线

上下文超过模型上限的 50% → 性能下降、指令遵守变差、可能改坏代码。**接近 50% 就果断 `/compact` 或开新会话**——不要等到满了再处理。新会话第一件事：让 AI 重读设计文档和任务文档。

### 2. 三回合止损

AI 同一问题上兜了三个回合还搞不定 → **停止**。不要在同一会话中反复重试——这会让 Session 充斥无效对话，AI 越试越偏。

正确做法：让它生成交接文档 → 切回问题出现前的状态 → 新 Session（最好换更高级模型）接手。

### 3. Fork Session ≠ 回滚代码

当 AI 工具调用反复出错、兜圈才理清误解时：开新会话，丢掉无效对话，**但保留代码改动**。回滚是撤销代码，Fork 只是重置上下文。

临时任务同理——新 Session 处理 → 切回原 Session。不让两件事混在一个上下文里。

## 七、Agent / 子代理策略

### Plan ↔ Build 循环

```
Plan（只读，设计方案）→ Build（读写，实现）
    ↑                      ↓
    └── 遇到问题切回分析 ←──┘
```

- **Plan Mode**：倾向向你提问而非自做决定（反正不用它动手），方案更详尽
- **Build（普通模式）**：倾向自己想办法，但可能擅自改需求——必须紧盯其输出，必要时打断

### @Explore 子代理：保护主上下文

```
❌ 主会话中全局搜索 → 大量无关文件污染上下文
✅ 用 Explore 子代理搜索 → 只把结论返回主会话
```

适用场景：调查代码使用情况、跨文件搜索、review（干净上下文更容易发现问题）。

### 委托更高级模型

当主力模型（DeepSeek/Haiku）卡住时，用 Opus/Sonnet 启动子代理做"专家咨询"——高成本模型只用在关键卡点。

## 八、文档管理 — 文档才是真成果

### 核心理念

代码只是副产品。文档（设计+任务+说明）才是持久资产——它让新 Session、新 Agent、甚至换 AI 工具时都能快速接手。

### 最小可行文档集

| 文档 | 作用 | 谁维护 |
|------|------|--------|
| **设计文档**（融合需求+设计） | 让 AI 提问来完善，保持更新 | Plan 产出，Build 中持续回写 |
| **任务文档**（融合方案+任务） | 精炼的上下文，追踪关键决策和进度 | 每完成一步让 AI 更新 |
| **说明文档**（CLAUDE.md） | 面向人和 AI，澄清事实避免误解 | 阶段变了立刻改 |

### 关键实践

- **让 AI 提问**来完善设计，而非你单向描述
- 每次任务完成后**让 AI 同步文档**
- 新会话第一件事：让 AI **重读设计文档和任务文档**
- AI 写的文件太长 → 让它精简；文档太多 → 让它整理 Wiki tree
- **对项目保持理解这件事你不能甩锅给 AI**

## 九、协作习惯

- 中文对话
- 每次修改后跑测试确认无回归
- 不确定的领域知识标记 `TODO(DOMAIN)` 不猜测
- 定期（每完成一个 milestone）回到学习 project 做经验总结
- **不要给 AI "参考意见"**：如果觉得重要就说是需求，拿不准就忍住不说。只要你说了，AI 大概率会遵守——不管你是不是用的虚拟语气

## 十、API 迁移（驱动 Claude 做旧→新迁移）

### 提示词五要素

```
1. 上下文  → 原脚本路径 + 功能简述（让 Claude 先读懂再动手）
2. 差异表  → 旧 API → 新 API 对照，每行带具体代码片段
3. 特殊陷阱 → 语义变了但编译器不报错的坑（负尺寸、隐式默认值消失）
4. 输出约束 → 新文件路径 + 格式
5. 验证标准 → 一个可量化的数值（频率、面积、多边形数），不能只写"运行不报错"
```

### 为什么需要"特殊陷阱"板块

`Rect(ref, dx, dy)` → `rectangle(ref, w, h)`：dx 可为负、w 必须为正。错误换算不报错但生成错误几何。**类型系统不保护的语义变更 = 最高优先级标注**。
