# playout 重写 — 可迁移经验

> 从 `D:\Project\playout` 芯片画图包重写实战中蒸馏的跨项目经验。持续更新。

---

## 1. CLAUDE.md 中的约定会被严格执行

playout 的 CLAUDE.md 写了 7 条核心约定（不可变优先、弧度制、Layer 集中管理等）。最终代码中 100% 得到了执行。

**结论**：投入时间写好 CLAUDE.md，它每次对话都在起作用。ROI 是指数级的。

---

## 2. Plan Mode 在项目开始时的价值最大

playout 重写一开始就进入 Plan Mode，产出了 398 行架构计划 + 8 Phase 依赖图。整个执行过程严格按照计划推进。

**结论**：Plan Mode 的最佳时机是项目刚开始、或大重构之前——此时架构决策最多、方向最不确定。

---

## 3. 测试精力应优先投放领域逻辑层

playout 重写初版中，core 层覆盖率 82-100%，但 component 层只有 13-22%。问题不是测试不够，而是测试集中在容易测的模块。

**结论**：在 Plan 中明确"按风险分配测试精力"，不要满足于"每个模块都有测试"。

---

## 4. 一次执行一个迭代，高风险改动独立提交

playout 优化阶段一次性执行了 P0+P1+P2 三迭代共 10 项改动。结果碰巧没问题（273 pass），但这不是好习惯——出了问题很难定位是哪项改动导致的。

**结论**：每迭代完成后跑一次测试。高风险改动绝不和大批量修改混在一起。

---

## 5. 依赖替换需要交叉验证

gdspy→gdstk、dxfgrabber→ezdxf、scipy.ellipk→mpmath.ellipk——三个替换看起来"API 差不多"，但每个都踩了坑：
- gdstk 旋转用弧度，gdspy 用度
- ezdxf 的 dxftype 是方法调用而非属性
- mpmath 返回高精度对象，需要 `float()` 转换

**结论**：依赖替换不能靠"看起来差不多"，必须写交叉验证测试。plan 中的"高风险区域"表格是一个好模式。

---

## 6. 代码审查报告是持续优化的有效驱动力

management 终端生成的 `03-review-and-optimization.md` 被 playout 终端直接读取并逐项执行。跨终端的分工让"审查"和"执行"可以交替进行。

**结论**：不要在一个终端里既写代码又审代码。分离关注点 → 换取客观性。

---

## 7. 可执行文档是"被动集成测试"——它发现的 Bug 单元测试永远找不到

playout 写用户指南时，为了验证代码示例"复制即跑"，执行了 Logo 加载、OAS 导出等真实路径，发现了 3 个问题：

| 发现 | 为什么 273 个单元测试没发现 |
|------|--------------------------|
| `poly.points *= 5.0` AttributeError | 从未有测试对只读 property 做 `*=` 赋值 |
| Logo `parents[2]` 路径错误 | Logo 的 `scale_to` 零测试覆盖 |
| IQASZ DXF 含 `*` 非法层名 | read_dxf 从未用真实 DXF 文件测试过 |

**结论**：单元测试覆盖的是"你想到要测的"，执行文档覆盖的是"用户真的会跑的"。两者之间的缝隙就是 Bug 藏身处。**未来每个项目都应要求文档中的代码块可自动验证**。

---

## 8. 集成测试揭露了 GDS 格式限制

集成测试构造 Chip + CPW + Xmon（79 多边形）→ 导出 OAS/GDS → 读回对比：

- OAS 往返：79→79，**精确一致**
- GDS 往返：79→86，多边形数增加了 7 个

根因：GDS 格式限制每个多边形最多 199 个顶点，超过的会被 gdstk 自动拆分。OAS 无此限制。

**结论**：集成测试的价值不仅是"验证正确"，更是**发现你完全不知道的格式约束**。如果只写单元测试，这个知识永远不会被触发。

---

## 9. API 迁移提示词的五要素结构

Purcell 滤波器脚本从旧版 playout 迁移到新版——~90 行脚本，5 个 API 差异点。使用以下五要素提示词，一次成功，谐振频率精确一致（6.346 GHz ≡ 6.346 GHz），287 tests 零回归：

```
1. 上下文  → 原脚本路径 + 功能简述
2. 差异表  → 旧 API → 新 API 精确对照（每行带代码片段，不写抽象描述）
3. 特殊陷阱 → 语义变更但编译器不报错的坑（负尺寸→正尺寸换算、隐式默认值消失）
4. 输出约束 → 新文件路径 + 格式
5. 验证标准 → 可量化的数值验证（频率、多边形数），不只"运行不报错"
```

### Rect→rectangle 是最危险的迁移类型

旧版 `Rect(ref, dx, dy)` 参数语义是"起点+延伸量"，dx/dy 可为负。新版 `rectangle(ref, w, h)` 是"左下角+宽高"，w/h 必须为正。编译器对此完全沉默——写反了不报错，但几何形状错误。

**教训**：迁移表中对每个"语义变更但类型系统不保护"的 API，必须单独列入陷阱板块，并给出换算公式。

### 验证标准必须量化

如果只写"确认运行不报错"，两个 Rect→rectangle 的错误可能被漏掉——因为错误换算下脚本照样跑通，只是几何形状错了。频率值 6.346 GHz 的精确验证才是真正的安全带。

**结论**：API 迁移的验证标准必须包含一个通过计算得到的数值（频率、面积、多边形数），不能只有布尔值（报错/不报错）。
