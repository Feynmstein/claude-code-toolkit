# OpenCode 经验 — 可迁移教训

> 来源：知乎 OpenCode 开发经验总结。以下为跨项目可迁移的教训，含 Claude Code 适配。

---

## 1. 上下文 50% 是硬红线

超过模型上限 50% → 性能下降、指令遵守变差、可能改坏代码。GitHub Copilot 和 Claude Code 都验证了这一点。**接近 50% 果断 `/compact`，不要等满了再处理。**

---

## 2. 三回合止损，不纠缠

同一问题 AI 兜了三回合还搞不定 → **停止**。不要在同一个 Session 中继续重试——无效对话积累会让 AI 越试越偏。

正确做法：让它写交接文档 → 回到问题前的状态 → 用新 Session（最好换更高级模型）接手。

---

## 3. 不要给 AI "参考意见"

这是最容易忽视的坑。你说"或许可以试试 X"，AI 就一定会去试 X——无论你语气多虚拟。**你觉得重要 → 说成需求；你拿不准 → 忍住不说**。这和 CLAUDE.md 编写中"约束 > 教程"是同一条原则。

---

## 4. 正面引导优于禁止

```
❌ "不要用 rm -rf"    → 上下文一长 AI 可能忘记甚至记反
✅ "清理文件时列出清单用 os.remove() 逐个处理。大量文件删除告诉我，我代劳"
```

---

## 5. Fork Session ≠ 回滚代码

AI 反复出错污染了上下文 → 开新会话，丢掉无效对话，**但保留代码改动**。回滚是撤销代码，Fork 只是重置上下文。新会话第一件事：让 AI 重读设计文档和任务文档。

临时任务同理：新 Session 处理 → 切回原 Session，不让两件事混在一个上下文里。

---

## 6. 用 @Explore 子代理保护上下文

```
❌ 主会话中全局搜索 → 无关文件污染珍贵上下文
✅ 用 Explore 子代理搜索 → 只把结论返回主会话
```

Review 也适合用子代理：干净上下文更容易发现问题，不会被主会话的固定思维限制。前提：设计文档和任务文档保持更新，否则新 Agent 无法快速理解项目。

---

## 7. 文档才是真成果，代码只是副产品

最精简的文档组合：**设计文档**（融合需求+设计）+ **任务文档**（融合方案+任务）+ **说明文档**（CLAUDE.md）。文档让新 Session、新 Agent、甚至换 AI 工具时都能快速接手。

每次任务完成后让 AI 同步文档。新会话第一件事：让 AI 重读关键文档。

---

## 8. Plan ↔ Build 循环

```
Plan（设计方案、只读）→ Build（写代码、读写）
    ↑                      ↓
    └── 遇到困难切回 Plan ←┘
```

Plan 倾向向你提问而非自做决定；Build 倾向自己想办法但可能擅自改需求——必须紧盯。

---

## 9. 你必须懂

> "如果 AI 用的是你不懂的技术，你就得先学会这个技术，至少有个大概了解。心智不匹配，工作就崩溃。"

对项目保持理解这件事，你不能甩锅给 AI。读完 AI 写的所有文档，太长就让它精简。
