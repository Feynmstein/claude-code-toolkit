## API 迁移提示词五要素

执行 API 迁移时，提示词按以下结构组织，能显著降低迁移出错率。

### 五要素模板

```
1. 上下文 — 原脚本路径 + 功能简述（让 Claude 先读懂要迁移什么）
2. 差异表 — 旧 API → 新 API 的精确对照（不要泛化，每个差异一行）
3. 特殊陷阱 — 语义变更而非签名变更的坑（符号反转、单位变化、类型变化）
4. 输出约束 — 新文件路径 + 格式 + 命名
5. 验证标准 — 可量化的验收条件（数值一致、可读回、目视检查）
```

### playout 验证

Purcell 滤波器脚本从旧版 playout 迁移到新版，使用此五要素提示词，结果：

- 谐振频率：旧版 6.346 GHz ≡ 新版 6.346 GHz（精确一致）
- 迁移对象：~90 行脚本，5 个 API 差异点
- 回归测试：287 passed，0 failures
- 一次成功，零返工

### 最容易出错的迁移类型

| 类型 | 示例 | 为什么危险 |
|------|------|-----------|
| 参数语义反转 | `Rect(ref, dx, dy)` dx/dy 可为负 → `rectangle(ref, w, h)` w/h 必须为正 | 编译器不报错，运行时生成错误几何 |
| 隐式默认值消失 | `Crossover(pos, angle=0)` W 隐式=4 → `Crossover(pos, W=4, angle_deg=0)` | 缺少新必需参数 → TypeError |
| 容器类型变化 | `Chip((0,0))` 作为通用容器 → `Structure()` | 语义对等但 API 完全不同 |
| 导出函数拆分 | `save_dxf(path)` 按后缀识别 → `write_oas()` / `write_dxf()` 显式选择 | 同一个方法名行为不同 |

### 防范措施

- 差异表中每行必须写出**旧版代码片段**和**新版代码片段**，不写抽象描述
- 对每个"语义变更但编译器不报错"的 API，单独列入"特殊陷阱"板块
- 验证标准必须包含一个**数值计算结果**（频率、面积、多边形数），不能只有"运行不报错"
